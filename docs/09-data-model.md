# Data Model — SelyoPass (MVP)

> Generated by the `architect` subagent from `06-system-design.md` (primary dependency), `03-prd.md`,
> and `idea.md` (source of truth). Scope is the June 30 Stellar Level 3 + APAC submission: **testnet
> only, Freighter only, Philippines only, anchor simulated, backend-less** (PRD §Non-goals; idea.md
> §10; system design §Components). Every entity/field traces to an `F-###` feature or `BR-###` rule.
>
> **Honesty constraints (do not skip):**
> - This is a **conceptual data model**, not a shipped schema. idea.md §9 (Known Limitations #9 in
>   project steering) states plainly: *"SEP-12 KYB extension is conceptual. Specification work pending
>   Level 3."* So concrete field names below — beyond the SEP-9 `organization.*` fields that idea.md §5
>   cites verbatim from the published standard, and the PH document categories idea.md §7/§1 names — are
>   **proposed, not authoritative**, and are tagged `[unverified]`.
> - The only **verified-from-published-standard** field names are the SEP-9 `organization.*` fields
>   listed in idea.md §5: `organization.name`, `organization.registration_number`,
>   `organization.registration_date`, `organization.director_name`, `organization.shareholder_name`,
>   and an incorporation-document image field. Everything else is grounded in the PH document
>   *categories* idea.md names (SEC, BIR, Mayor's Permit, Articles of Incorporation, beneficial
>   ownership / UBO, GIS) but the exact field encodings are proposed.
> - **No data-retention numbers, encryption specifics, or storage limits are invented.** idea.md and
>   the PRD set none. They are promoted to OPEN QUESTIONS, not guessed.
> - There is **no application database** in the MVP (system design: backend-less, non-custodial). The
>   "data model" is split across **on-chain contract state** and **off-chain startup-held documents**.
>   There are no SQL tables to define — contrast the ReconLens example (different domain, has a DB).

## Entities & relationships (ERD)

The model has three core entities, split across the on-chain / off-chain trust boundary. The
`KybCredential` (the extended SEP-12/SEP-9 record, F-001) is the logical payload; the
`OnChainCredentialRecord` (F-002) is what the Soroban contract actually persists on testnet; the
`OffChainDocument` (F-003) is what the startup retains. The on-chain record holds only **hashes** of
the off-chain documents — never the documents themselves (BR-002, BR-004).

```
         off-chain (startup-held, NON-custodial — BR-002/BR-004)        on-chain (Soroban testnet)
   ┌───────────────────────────────────────────────────┐      ┌──────────────────────────────────────┐
   │  OffChainDocument  (F-003)                          │      │  OnChainCredentialRecord (F-002)       │
   │  - the actual SEC / BIR / Mayor's Permit / AoI /    │      │  - credential_id                       │
   │    GIS / UBO files (PII-bearing — RA 10173/BR-006)  │      │  - issuer/anchor signature             │
   │  - content_hash  ──────────────────────────────────┼─────►│  - anchor_public_key (ref)             │
   │    (sha-256 of file bytes) [unverified algo]        │ 1..* │  - document_hash[]  (anchored hashes)  │
   └───────────────────────────────────────────────────┘ anchors  - status                              │
                       ▲                                  └─────────│  - structured KYB fields (F-001)       │
                       │ recompute & compare (F-004, UJ-002 step 4) └──────────────────────────────────────┘
                       │                                                          ▲
   ┌───────────────────┴───────────────────────────────┐                         │ serializes / embeds
   │  KybCredential  (F-001 — the extended SEP-12/SEP-9  │ ────────────────────────┘
   │  schema; logical payload, conceptual [unverified])  │
   │  = SEP-9 organization.* (published)                 │
   │    + PH extension fields (F-001 contribution)       │
   └─────────────────────────────────────────────────────┘

   Cardinality: one KybCredential ──issued-as──► one OnChainCredentialRecord.
                one OnChainCredentialRecord ──anchors──► 1..* document_hash (one per OffChainDocument).
                one OffChainDocument ──has──► exactly one content_hash.
```

> N/A — no `Account`/`Period`/`User` table hierarchy as in the ReconLens example, **because** SelyoPass
> is backend-less and wallet-authenticated; identity is a Stellar account + the anchor keypair, not a
> stored user row (system design §Components, "no application database").

## Schema / field definitions

> **Legend.** Location = `on-chain` (Soroban contract state) / `off-chain` (startup-held) / `derived`
> (computed client-side, not stored). PII = carries personal data of officers / beneficial owners
> (RA 10173 / BR-006 relevant). Req = required vs optional for a valid credential. Field names tagged
> `[unverified]` are proposed encodings, not from a published spec (idea.md §9: schema is conceptual).

### KybCredential (F-001 — extended SEP-12/SEP-9 KYB payload; conceptual `[unverified]`)

This is the logical credential payload. It has two layers: (a) the **published SEP-9 `organization.*`
fields** idea.md §5 cites, and (b) the **F-001 PH extension fields** — the actual contribution idea.md
names (SEC, BIR, Mayor's Permit, Articles of Incorporation, UBO, GIS). Per idea.md §5, the SEP-9 org
fields are "thin: no beneficial-ownership percentage, no GIS, no Mayor's Permit, no BIR" — the
extension fills exactly that gap.

**(a) Published SEP-9 `organization.*` fields** (names verified from idea.md §5 / SEP-9):

| Field | Type | Location | PII | Req | Description |
|-------|------|----------|-----|-----|-------------|
| organization.name | text | on-chain (structured field) | PII (low) | yes | Legal entity name. From SEP-9. |
| organization.registration_number | text | on-chain | PII (low) | yes | Entity registration number. From SEP-9. |
| organization.registration_date | date | on-chain | no | yes | Date of registration. From SEP-9. |
| organization.director_name | text | on-chain | **PII** | yes | Director name. From SEP-9. |
| organization.shareholder_name | text | on-chain | **PII** | yes | Shareholder name. From SEP-9. |
| organization.*_incorporation_doc (image) | document → hashed | off-chain (doc) + on-chain (hash) | **PII** | yes | Incorporation-document image. SEP-9 defines an org incorporation-doc image field; in SelyoPass the **image stays off-chain, only its hash is anchored** (F-003, BR-002). Exact SEP-9 field name `[unverified]`. |

**(b) F-001 PH extension fields** (the F-001 contribution — categories from idea.md §1/§7; encodings
`[unverified]` because the schema is conceptual per idea.md §9):

| Field | Type | Location | PII | Req | Description |
|-------|------|----------|-----|-----|-------------|
| sec_registration `[unverified]` | document + structured ref | off-chain (doc) + on-chain (hash + maybe number) | **PII** | yes | SEC registration. **F-001 extension.** Document hashed off-chain; a structured SEC number may also be carried (Level 1 demo used a `SelyoPass:{SEC#}` memo per steering — but that is the rejected payment-memo path, not F-002 issuance). |
| bir_certificate `[unverified]` | document | off-chain (doc) + on-chain (hash) | **PII** | yes | BIR certificate. **F-001 extension** (absent from SEP-9 per idea.md §5). |
| mayors_permit `[unverified]` | document | off-chain (doc) + on-chain (hash) | **PII** | yes | Mayor's Permit. **F-001 extension** (absent from SEP-9). |
| articles_of_incorporation `[unverified]` | document | off-chain (doc) + on-chain (hash) | **PII** | yes | Articles of Incorporation. **F-001 extension** (SEP-9 has a thin incorporation-doc image; this is the fuller PH artifact). |
| general_information_sheet (GIS) `[unverified]` | document | off-chain (doc) + on-chain (hash) | **PII** | yes | GIS. **F-001 extension** (absent from SEP-9). |
| beneficial_owners[] / UBO `[unverified]` | list of records | off-chain (doc) + on-chain (hash; possibly structured) | **PII (sensitive)** | yes | Ultimate beneficial ownership disclosure. **F-001 extension.** MUST align to **SEC Memorandum Circular No. 15, s.2025** tightened UBO disclosure (BR-005), effective Jan 2026. Likely includes owner identity + ownership percentage (idea.md §5 notes SEP-9 lacks "beneficial-ownership percentage"). Exact UBO sub-fields `[unverified]` — pending the conceptual schema being specified. |

> **UBO sub-fields (proposed, `[unverified]`, BR-005):** per-owner `name`, `ownership_percentage`, and
> identity reference are implied by idea.md §5 ("no beneficial-ownership percentage") and BR-005. The
> precise field list required by SEC MC No. 15 s.2025 is **not enumerated in idea.md** — flagged as an
> OPEN QUESTION rather than invented.

### OnChainCredentialRecord (F-002 — Soroban contract state on testnet)

What the Soroban contract actually persists. This is the authoritative on-chain record (system design:
"the authoritative on-chain credential record and anchored hashes"). All fields are **on-chain**.

| Field | Type | PII | Req | Description |
|-------|------|-----|-----|-------------|
| credential_id `[unverified]` | identifier | no | yes | Credential identifier. Named in task brief; concrete type/format `[unverified]`. |
| anchor_signature | signature bytes | no | yes | The (simulated) anchor's signature over the credential. F-002 acceptance: "the issued credential carries the anchor's signature." Verified by reader against the anchor public key (F-004, BR-001). |
| anchor_public_key (reference) | Stellar public key / ref | no | yes | Reference to the anchor's public key, against which F-004 verifies the signature (BR-001 — anchor is verifier of record). |
| document_hash[] | list of content hashes | no | yes | One content hash per anchored document (F-003). **The documents themselves are never stored on-chain** (BR-002). Hash algorithm `[unverified]` (idea.md/PRD do not pin it; system design says "content hash" without algorithm). |
| status `[unverified]` | enum | no | yes | Credential status/identifier state. Named in task brief; concrete states `[unverified]` (idea.md/PRD do not enumerate; see OPEN QUESTIONS on revocation/lifecycle). |
| structured KYB fields (F-001) | embedded / serialized | mixed (see F-001 table) | yes | The structured `organization.*` + non-document extension fields the reader reads (F-004 step 2; UJ-002). Whether stored inline in contract state or referenced is a contract-design detail `[unverified]`. |
| deployment_address | contract address | no | yes (artifact) | Soroban contract deployment address — a required submission artifact (idea.md §8, F-002 acceptance). Property of the deployed contract, not a per-credential field. |
| interaction_tx_hash | tx hash | no | yes (artifact) | Issuance interaction transaction hash — required submission artifact (idea.md §8, F-002 acceptance). |

### OffChainDocument (F-003 — startup-held, non-custodial)

The actual document files. **Off-chain, retained by the startup** (BR-002, BR-004). SelyoPass never
custodies these.

| Field | Type | Location | PII | Req | Description |
|-------|------|----------|-----|-----|-------------|
| document_bytes | binary file | off-chain | **PII (sensitive)** | yes | The actual SEC/BIR/Mayor's Permit/AoI/GIS/UBO file. Held by the startup (BR-004). Storage mechanism **under-specified** — idea.md §3/§7 say "the startup retains documents (e.g., locally)"; steering floats IPFS but idea.md does not commit. `[unverified]` |
| content_hash | hash value | derived → anchored on-chain | no | yes | sha-256-style content hash of `document_bytes`, computed client-side (system design data flow UJ-001 step 3). This is the only part that goes on-chain (F-003, BR-002). Algorithm `[unverified]`. |
| document_type `[unverified]` | enum | off-chain | no | yes | Which PH document this is (SEC / BIR / Mayor's Permit / AoI / GIS / UBO). Proposed; encoding `[unverified]`. |

> **On-chain vs off-chain, stated explicitly (task requirement 2 / BR-002):** ON-CHAIN = the
> `OnChainCredentialRecord` (id, anchor signature, anchor public key reference, document hashes, status,
> structured non-document KYB fields, deployment address, interaction tx hash). OFF-CHAIN = every
> actual document (`OffChainDocument.document_bytes`) and the PII they contain. The boundary is
> load-bearing: putting a document on-chain would violate BR-002/BR-004 and leak RA 10173-regulated
> personal data (BR-006) — explicitly the rejected alternative in system design §Key technology choices.

## Constraints & indexes

> No relational DB exists, so these are **contract-state and integrity constraints**, not SQL
> keys/indexes.

- **Hash-match integrity (F-003, F-004, BR-002).** A reader recomputes `content_hash` from the
  startup-presented document and compares it to the anchored `document_hash`. Match ⇒ integrity
  confirmed; mismatch ⇒ reject with a clear reason (PRD F-004 acceptance; UJ-002 step 4). This is the
  core integrity constraint of the model.
- **Signature-verification constraint (F-004, BR-001).** `anchor_signature` MUST verify against
  `anchor_public_key`; a bad signature ⇒ reject (PRD F-004 acceptance). The anchor is the verifier of
  record (BR-001) — for MVP the anchor is **simulated** on testnet (`[unverified]`, idea.md §10).
- **Required-field validation (F-001, BR-005).** A credential missing a required field is rejected with
  a field-level error (PRD F-001 acceptance), including the UBO fields mandated by SEC MC No. 15 s.2025
  (BR-005).
- **No-custody constraint (BR-002, BR-004).** No `document_bytes` may be written to contract state — only
  hashes. This is an invariant of the whole design, not a tunable.
- **Append-only / immutability of an issued record — `[unverified]`/OPEN QUESTION.** The task asks
  whether an issued credential record is append-only/immutable. idea.md and the PRD **do not state**
  this guarantee. On-chain records are practically immutable once written, but whether the contract
  supports update/revocation, or treats issuance as append-only, is **not specified** — flagged below,
  not asserted. (Contrast the ReconLens example's explicit append-only `Resolution` table — that
  guarantee was stated upstream there; here it is not.)
- **Indexes:** N/A — **because** there is no queryable datastore; reads are contract-state lookups by
  `credential_id` on testnet (system design §Integration points).

## Retention & privacy classification

- **PII inventory (RA 10173 / BR-006).** The following carry personal data of officers / beneficial
  owners: `organization.director_name`, `organization.shareholder_name`, the incorporation-doc image,
  and the off-chain documents (`sec_registration`, `bir_certificate`, `mayors_permit`,
  `articles_of_incorporation`, `general_information_sheet`, `beneficial_owners`/UBO). UBO data is the
  most sensitive (identity + ownership of natural persons). Per BR-006, SelyoPass is a
  **personal-information controller under RA 10173**, crossing the NPC 1,000-individual threshold;
  NPC registration + a privacy management program are required **before federating any real corporate
  data**. For the testnet MVP **no real corporate data is federated** (`[unverified]`, PRD BR-006).
- **Classification.** On-chain hashes, signatures, public-key reference, and `credential_id`/`status`
  are **non-PII** (a hash does not reveal document contents). The **PII lives off-chain** in the
  documents. This is the privacy rationale for the hash-only design (BR-002).
- **Retention — OPEN QUESTION, not invented.** idea.md and the PRD specify **no retention period, no
  deletion policy, and no right-to-erasure flow** for off-chain documents. Note that on-chain hashes
  are effectively permanent on the ledger, which interacts awkwardly with any future RA 10173 erasure
  obligation — flagged below. **No numbers are invented here.**
- **Encryption at rest — OPEN QUESTION, not invented.** No encryption-at-rest requirement for the
  off-chain document store is stated anywhere in idea.md/PRD. Flagged below.
- **Contrast with example:** the ReconLens model could state "encrypted at rest; deleted with the
  account" because its upstream docs supported it. SelyoPass's upstream docs do **not**, so those
  statements are NOT made here — they are open questions.

## Migration notes

- **No schema migrations in the MVP sense — `[unverified]`/N/A.** There is no relational DB to migrate
  (backend-less). The two evolving surfaces are (a) the **conceptual SEP-12/SEP-9 extension schema**
  (F-001), which idea.md §9 marks as not-yet-specified, and (b) the **Soroban contract** structure.
- **Schema evolution risk (F-001, BR-005).** Because the KYB schema is conceptual and must track
  **SEC MC No. 15 s.2025** UBO changes (effective Jan 2026, BR-005), the field set is expected to
  change. The "extend a published standard" stance (idea.md §5/§6) means the SEP-9 `organization.*`
  base should stay aligned with upstream SEP-9; the PH extension fields are the volatile part.
- **Forward-looking (final vision, out of MVP scope):** idea.md §7 F-102 anticipates cross-jurisdiction
  schema extension (Indonesia OJK-compatible UBO fields, Bahasa-language fields, jurisdiction-tagged
  hashes). Steering §"Implications" #4 says design the credential for cross-jurisdiction extensibility
  from day one — relevant to how additive the extension schema should be, but **not built in MVP**.
- **Contract migration / upgradeability — OPEN QUESTION.** Whether the Soroban contract is upgradeable,
  and how an issued record's structure could change post-deployment, is not specified in idea.md/PRD.
  Flagged below.

## Open questions (under-specified data NFRs — not converted into guesses)

- **Document content-hash algorithm is not pinned.** System design says "content hash"; idea.md/PRD
  name no algorithm (sha-256 is assumed but `[unverified]`).
- **Off-chain document storage mechanism + access control is under-specified.** "Startup retains (e.g.,
  locally)" vs IPFS; if networked, who may fetch a PII-bearing document is undefined and
  security-sensitive (BR-006). `[unverified]` / `[open]`
- **Retention / deletion / RA 10173 erasure is unspecified.** No retention period or deletion flow for
  off-chain documents; on-chain hash permanence conflicts with erasure obligations (BR-006). `[open]`
- **Encryption-at-rest for off-chain documents is unspecified.** `[open]`
- **Issuer/anchor key storage and custody is unspecified and security-critical.** A leaked issuer key
  forges credentials (carried from system design). `[open]`
- **Credential `status` lifecycle is undefined.** Are issued records append-only/immutable? Is
  revocation or re-issuance supported? idea.md/PRD do not say — do **not** assume immutability as a
  feature. (F-101 "continuous credentials" is final-vision, not MVP.) `[open]`
- **Exact UBO sub-field list per SEC MC No. 15 s.2025 is not enumerated** in idea.md (only that the
  schema "must carry current UBO fields", BR-005). `[open]`
- **The SEP-12/SEP-9 extension field encodings are conceptual** (idea.md §9), so all `[unverified]`
  field names above are proposed, not authoritative.
- **Whether structured KYB fields live inline in contract state or are referenced** is a contract-design
  detail not specified upstream. `[unverified]`

## Proposed ADRs

1. **ADR — Anchor only document hashes on-chain; keep all documents and PII off-chain (non-custodial
   data boundary).** (BR-002, BR-004, BR-006; F-003)
2. **ADR — Model the F-001 credential as published SEP-9 `organization.*` fields plus a PH extension
   layer, rather than a bespoke schema.** (F-001; idea.md §5/§6)
3. **ADR — Define the UBO sub-schema to satisfy SEC MC No. 15 s.2025, and the policy for keeping it
   aligned as the regulation evolves.** (BR-005)
4. **ADR — Decide the credential lifecycle/status model (append-only vs revocable/updatable) for the
   on-chain record.** (open question; relates to F-101 future scope)
5. **ADR — Choose the off-chain document storage + access-control mechanism (local vs IPFS) given
   RA 10173 obligations.** (F-003, BR-006)
