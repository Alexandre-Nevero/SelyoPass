## Data classification

Pulled from the data model (`09-data-model.md` §Retention & privacy classification). The
load-bearing privacy control is the **hash-only design (BR-002)**: PII lives off-chain; only non-PII
hashes/signatures go on-chain.

| Class | What | Where it lives | Notes |
|-------|------|----------------|-------|
| **PII (sensitive)** | UBO / beneficial-ownership records (identity + ownership % of natural persons) | **off-chain** document (`OffChainDocument.document_bytes`) | Most sensitive class. SEC MC No. 15 s.2025 (BR-005). RA 10173-regulated (BR-006). |
| **PII** | SEC registration, BIR certificate, Mayor's Permit, Articles of Incorporation, GIS files; `organization.director_name`, `organization.shareholder_name`, incorporation-doc image | **off-chain** documents (PII bytes) + structured names embedded **on-chain** | Document bytes stay off-chain (BR-002, BR-004); director/shareholder *names* are structured fields the reader reads (F-004) and may sit on-chain — see OPEN QUESTION on on-chain PII minimization. |
| **Non-PII (public by design)** | `document_hash[]`, `anchor_signature`, `anchor_public_key` reference, `credential_id`, `status`, `deployment_address`, `interaction_tx_hash` | **on-chain** Soroban contract state (testnet) | A content hash does not reveal document contents — this is the privacy rationale for BR-002. Intentionally publicly readable (F-004 is a public reader). |
| **Secret (security-critical)** | Simulated anchor issuer **private key** | **undefined** (custody unspecified upstream) | A leaked issuer key forges credentials. Custody/exposure is an OPEN QUESTION (system design + data model both flag it). |
| **Secret (user-held, never ours)** | The Freighter user's private key | inside the Freighter extension, user-side | SelyoPass never handles it (BR-002 non-custodial posture; F-006). |

> N/A — **no application database, no object-storage service, no stored auth identity (email), no
> bank/processor credentials**, because the MVP is backend-less and wallet-authenticated (system
> design §Components: "no application database"). This is the opposite of the ReconLens example,
> which classified DB rows and an auth-provider email — different domain.

> **On-chain PII caveat (flagged, not resolved):** the data model carries some structured PII
> (`director_name`, `shareholder_name`, and possibly inline structured KYB fields) on-chain so the
> reader can read them (F-004 step 2). On-chain data is effectively permanent. Whether structured PII
> should be minimized / kept off-chain alongside the documents is **unspecified upstream** — see
> OPEN QUESTIONS (this interacts directly with the RA 10173 erasure gap).

## Authn / authz model

There is **no SelyoPass-operated server in the MVP**, so there is no server-side auth layer to
design (system design §Context diagram). The trust boundary is the **Soroban contract** and the
**wallet**. Each network-exposed surface is enumerated below with its auth/authz posture stated
explicitly. **Where the posture is undefined, it is flagged loudly** — it is not silently assumed.

### Per-surface auth/authz posture

- **Soroban credential contract (publicly callable on testnet) — F-002.**
  **Posture: UNDER-SPECIFIED → OPEN QUESTION + proposed ADR.** The contract is publicly callable on
  testnet. The *intended* model, inferred from business rules (not asserted as implemented):
  - **Issuance (`issue`) MUST be gated to the anchor key.** BR-001 makes the anchor the verifier of
    record; only the (simulated) anchor key may mint a credential. The concrete enforcement
    mechanism (e.g., `require_auth` on the issuer, an allow-listed issuer key) is **not specified
    upstream** — flagged. If issuance is left open, **anyone can forge a credential** (see Elevation
    / Spoofing threats).
  - **Read / verify is intentionally public.** F-004 is a public relying-party reader; reading
    credential state, the anchored hashes, and the anchor public key is open by design.
  This split (gate writes to the anchor, keep reads public) is a **proposed ADR**, mirroring system
  design §Open questions. It is the single most important authorization decision in the system and
  is **currently undefined in code**.

- **Freighter wallet API — F-006.**
  **Posture: user-side auth (signing consent); private key never handled by the app.** The wallet
  *is* the user-side authentication: a state-changing transaction (issuance) requires the user's
  explicit signing consent in Freighter. SelyoPass does not, and must not, touch the private key
  (BR-002 non-custodial). Missing extension / declined signature / wrong network are first-class
  error states (F-006). This is the one auth surface that is **well-defined** upstream.

- **Horizon / Soroban RPC (public testnet endpoints).**
  **Posture: public, no credentials.** These are public Stellar testnet endpoints
  (`horizon-testnet.stellar.org` verified in steering; Soroban RPC endpoint not pinned in repo
  config, `[unverified]`). No authentication is expected or designed. Risk is availability /
  rate-limiting, not access control (see DoS threat).

- **Off-chain document retrieval path — F-003.**
  **Posture: ACCESS CONTROL UNDEFINED → OPEN QUESTION + proposed ADR (security-sensitive).** The
  storage/retrieval mechanism is under-specified upstream — idea.md §3/§7 say only "the startup
  retains documents (e.g., locally)"; steering floats IPFS but idea.md does not commit. **Auth/authz
  cannot be designed until the mechanism is chosen.** If any networked store (e.g., IPFS) is used,
  *who may fetch a PII-bearing document, and how that is controlled, is undefined* — and the payload
  is RA 10173-regulated personal data (BR-006). This is the highest-risk undefined surface for
  information disclosure.

- **Simulated anchor issuer key.**
  **Posture: whoever holds the key can mint credentials → custody UNSPECIFIED, security-critical.**
  For a demo this may be an env-held testnet key, but its generation, storage, and exposure are
  **unspecified upstream**. A leaked issuer key forges credentials (Elevation threat). Flagged as an
  OPEN QUESTION in both system design and data model; carried here as the top secrets-handling gap.

> N/A — **no role hierarchy, no per-account row scoping, no session management**, because there is no
> server and no stored user table (contrast ReconLens, which scoped every query to an authenticated
> account). Identity here is a Stellar account + the anchor keypair, not a stored user row.

## Threat model

STRIDE-style pass tied to the network-exposed surfaces above. Mitigations are mapped to `F-###` /
`BR-###` **where they exist upstream**; where no control is specified, the cell says **GAP / OPEN
QUESTION** rather than inventing a mitigation.

| Threat | Surface / Vector | Impact | Mitigation (or GAP) |
|--------|------------------|--------|---------------------|
| **Spoofing** | Forged anchor signature on a credential; or a non-anchor caller invoking `issue` on the publicly callable contract | A relying party trusts a credential the real anchor never issued | **Partial:** F-004 verifies `anchor_signature` against `anchor_public_key` (BR-001) — a bad signature is rejected. **GAP:** issuance authorization (gate `issue` to the anchor key) is **undefined** (BR-001 implies it, not implemented) → OPEN QUESTION + ADR. Spoofing also collapses into Elevation if the issuer key leaks. |
| **Tampering** | Altering an off-chain document after issuance so it no longer matches the anchored hash | Relying party could be shown a doctored document | **Mitigated by design:** F-003/F-004 hash-match constraint — reader recomputes `content_hash` and compares to the anchored `document_hash`; mismatch ⇒ reject with a clear reason (BR-002). This is the core integrity control. **Sub-GAP:** hash algorithm is not pinned upstream (`[unverified]`; sha-256 assumed). |
| **Tampering** | Mutating on-chain contract state | Corrupted credential record | On-chain records are practically immutable once written. **GAP:** whether the contract is append-only vs updatable/revocable, and whether it is upgradeable, is **unspecified** (data model OPEN QUESTION) — do not assert immutability as a designed control. |
| **Repudiation** | "The anchor never issued this" / "I didn't sign that" | Dispute over credential provenance | **Partial:** every credential carries `anchor_signature` + `anchor_public_key` reference (F-002, F-004), and the issuance `interaction_tx_hash` is an on-chain artifact (idea.md §8) — chain state is the audit record. **GAP:** no application-level audit log exists (backend-less); credential lifecycle/`status` semantics are undefined. |
| **Information disclosure** | PII leak via the off-chain document retrieval path (esp. UBO data) if a networked store has no access control | Exposure of officers'/beneficial owners' personal data — RA 10173 / BR-006 breach | **Partial (design):** hash-only on-chain design (BR-002) keeps PII *off* the public ledger. **GAP:** off-chain document access control is **undefined** (mechanism not chosen) → OPEN QUESTION + ADR. **GAP:** no encryption-at-rest requirement for off-chain docs is specified. |
| **Information disclosure** | Structured PII (`director_name`, `shareholder_name`) embedded on the public, permanent ledger | Permanent public exposure of named individuals | **GAP:** on-chain PII minimization is unspecified upstream → OPEN QUESTION (interacts with RA 10173 erasure — on-chain hashes/fields can't be deleted). |
| **Denial of service** | Testnet / Horizon / Soroban RPC unavailability, rate-limiting, or chain congestion | Issuance or verification can't complete during a demo | **Partial:** F-006 mandates explicit loading/error states for RPC/issuance/verification failures rather than silent crashes. **GAP:** no availability/latency target, no retry/rate-limit strategy (backend-less, nowhere to enforce limits) — system design flags this as an OPEN QUESTION. Demo-scale only; no production load claimed. |
| **Denial of service** | Oversized / malformed document submitted client-side | Browser exhaustion | **GAP:** no upload size/row cap is specified upstream (contrast ReconLens' explicit cap). Client-side only; flagged, not invented. |
| **Elevation of privilege** | **Issuer-key compromise** — attacker obtains the simulated anchor private key | Attacker mints arbitrary valid-looking credentials; the entire trust model collapses | **GAP (security-critical):** issuer-key custody/storage is **unspecified** → top OPEN QUESTION + proposed ADR. No rotation, no HSM/secret-store requirement stated upstream. This is the highest-severity gap because a single leaked key forges everything. |
| **Elevation of privilege** | Unauthorized caller invokes contract `issue` because the contract is publicly callable | Credential forgery without the anchor key | **GAP:** contract authorization model undefined (same as Spoofing row) → OPEN QUESTION + ADR. |

## Compliance obligations

Grounded **strictly in idea.md §9** (and BR-001..BR-006 in the PRD). All `said`-grade desk research,
not legal advice — `[assumption — confirm with counsel before any real-data launch]`. Sources are
the ones idea.md cites; **no standards or numbers are invented here.**

| Obligation | Source (idea.md §9) | What it requires | MVP applicability |
|------------|---------------------|------------------|-------------------|
| **RA 10173 (Data Privacy Act) + NPC Circular 2022-04** | privacy.gov.ph via bakermckenzie.com | SelyoPass is a **personal-information controller** (collects officer / beneficial-owner data). Requires a **DPO + Data Processing System registration** for entities processing personal data of 250+ employees, **1,000+ individuals**, or in a risk-posing way — SelyoPass crosses the 1,000-individual threshold quickly. Implies **NPC registration + a privacy management program**. Administrative fines up to **3% of annual gross income** (NPC Circular 2022-01). | **PRE-REAL-DATA GATE, not an MVP runtime control.** For the testnet MVP **no real corporate data is federated** (BR-006, `[unverified]`). NPC registration + privacy program are required **before federating any real corporate data** — a hard gate for anything beyond the demo. |
| **BSP Circular 1170 (30 Mar 2023)** | lexology.com | Permits reliance on third-party / digital-ID systems **but the relying covered person retains ultimate responsibility** for identification and verification. **Liability is non-transferable.** | **Applies to product framing now (BR-003).** The credential must be framed as structured, signed, hash-verifiable document *transmission* — **never** a "verified, trust-this-business" stamp. F-004/BR-003: the reader presents accept/reject **without asserting compliance approval**. This is a runtime UI constraint even in the MVP. |
| **SEC Memorandum Circular No. 15, s.2025** | verihubs.com/blog/kyb-know-your-business | Tightened **ultimate-beneficial-owner (UBO) disclosure**, effective **January 2026**. | **Applies to schema design now (BR-005).** The F-001 KYB schema must carry current UBO fields. **GAP:** the exact UBO sub-field list required by the circular is **not enumerated upstream** (data model OPEN QUESTION) — flagged, not invented. |

**The MVP-vs-pre-real-data boundary (stated explicitly):**
- **In MVP scope now (runtime):** BR-003 liability framing (BSP 1170) enforced in the reader UI;
  BR-005 UBO-field alignment in the conceptual schema (SEC MC No. 15 s.2025); the hash-only
  non-custodial data boundary (BR-002).
- **Pre-real-data gate (NOT MVP runtime):** the full RA 10173 / NPC Circular 2022-04 obligation
  (NPC registration, DPO, privacy management program). Because **no real corporate data is federated
  on testnet** (BR-006, `[unverified]`), this is a launch gate before real data, not a control the
  MVP demo implements.

## Secrets handling

- **Simulated anchor issuer private key — the load-bearing secret.** Whoever holds it can mint
  credentials (Elevation threat). **Custody, generation, storage, and rotation are UNSPECIFIED
  upstream** → OPEN QUESTION + proposed ADR. The principle (never inline a key in code, config, or
  logs) is asserted as a *requirement*, but **no concrete storage mechanism (env var vs secret store
  vs HSM) is mandated upstream** — that gap is flagged, not filled with an invented control.
- **Freighter user private key.** Never handled by SelyoPass — it stays inside the extension; signing
  happens user-side (BR-002, F-006). Nothing for us to store or rotate.
- **No other secrets exist in the MVP** — N/A for auth-provider keys, DB credentials, storage
  credentials, or third-party API keys, because the system is backend-less with no DB, no object
  store, and no integrated third-party APIs (system design §Components / §Integration points). If
  the off-chain store later becomes a networked service (e.g., IPFS gateway with auth), its
  credentials become a new secret to manage — flagged with the off-chain mechanism OPEN QUESTION.

## Audit & logging

- **On-chain artifacts are the de facto audit record.** The issuance `interaction_tx_hash` and the
  `deployment_address` (idea.md §8, F-002 acceptance), plus the on-chain `anchor_signature` and
  `anchor_public_key` reference, provide a tamper-evident provenance trail for who issued what
  (supports the Repudiation mitigation).
- **No application-level audit log exists** in the MVP, because it is backend-less — N/A with cause.
  There is no server to record auth events or actions (contrast ReconLens, which logged auth +
  resolution actions).
- **What must NOT be logged / put on-chain (assertable from BR-002 / BR-006):** document bytes and
  their PII must never be written on-chain (BR-002) and must never be echoed into client logs or
  error messages (F-006 errors should be legible without leaking PII). The anchor private key must
  never appear in logs.
- **GAP:** there is no defined client-side logging policy, and the credential `status`/lifecycle is
  undefined — so there is no specified audit trail for revocation/re-issuance events. Flagged, not
  invented.

## Incident response basics

Lightweight and honest about the backend-less constraint — most "response" is forward-only because
on-chain state is effectively immutable.

- **Detection.** No monitoring/alerting is specified upstream (backend-less) → GAP / OPEN QUESTION.
  The practically observable signals are RPC/issuance/verification failures surfaced by F-006 and
  on-chain inspection of contract state.
- **Issuer-key compromise (highest-severity scenario).** If the simulated anchor key leaks, every
  credential it signed is suspect. **There is no specified rotation or revocation path** (key
  custody and credential `status`/revocation are both OPEN QUESTIONS). Forward-only recovery would
  mean rotating to a new anchor key and re-issuing — but the mechanism is undefined upstream.
  Flagged as a critical gap, not papered over.
- **Off-chain PII exposure.** If the off-chain retrieval path leaks documents (RA 10173 / BR-006),
  the response is a regulatory/NPC matter — but for the testnet MVP **no real corporate data is
  federated** (BR-006), which bounds the blast radius to demo data. Pre-real-data, the NPC
  privacy-management program would define breach response — that program is the pre-launch gate, not
  an MVP control.
- **Tampering / bad credential.** A doctored document is caught at read time by the hash-match
  constraint (F-003/F-004) and rejected — recovery is "reject and re-present a matching document,"
  not a rollback.
- **Rollback.** On-chain records are practically immutable; recovery is **forward-only** (issue a
  corrected record), and whether the contract even supports a corrective/revocation update is an
  OPEN QUESTION.

## Open security questions (under-specified — NOT converted into guesses)

1. **Soroban contract authorization model is undefined.** The contract is publicly callable on
   testnet; who may invoke `issue` (anchor-gated, per BR-001) vs read/verify (public, per F-004)
   needs an explicit, enforced rule. `[open]`
2. **Simulated-anchor issuer-key custody/storage/rotation is unspecified and security-critical.** A
   leaked issuer key forges all credentials. `[open]`
3. **Off-chain document storage mechanism + access control is under-specified.** Local vs IPFS; if
   networked, who may fetch a PII-bearing document is undefined and RA 10173-sensitive (BR-006).
   `[open]` / `[unverified]`
4. **Encryption-at-rest for off-chain documents is unspecified.** `[open]`
5. **Retention / deletion / RA 10173 right-to-erasure is unspecified**, and on-chain hash (and
   possibly on-chain structured PII) permanence conflicts with any future erasure obligation.
   `[open]`
6. **On-chain PII minimization is unspecified** — whether `director_name`/`shareholder_name` /
   structured KYB fields should stay off-chain rather than on the permanent public ledger. `[open]`
7. **Document content-hash algorithm is not pinned** (sha-256 assumed; integrity control depends on
   it). `[unverified]`
8. **Credential `status`/lifecycle (append-only vs revocable/updatable) and contract upgradeability
   are undefined** — no revocation path for a compromised or superseded credential. `[open]`
9. **No availability / DoS posture** (no targets, no retry/rate-limit strategy) for the
   testnet/Horizon/Soroban RPC dependency. `[open]`
10. **Exact UBO sub-field list per SEC MC No. 15 s.2025 (BR-005) is not enumerated upstream.**
    `[open]`

## Proposed ADRs

1. **ADR — Define the Soroban contract authorization model: gate `issue` to the anchor key, keep
   read/verify public.** (BR-001, F-002, F-004)
2. **ADR — Choose issuer-key custody, storage, and rotation for the simulated anchor (testnet
   secret handling), treating issuer-key compromise as the top threat.** (BR-001; secrets handling)
3. **ADR — Choose the off-chain document storage + access-control mechanism (local vs IPFS) and its
   encryption-at-rest posture, given RA 10173 obligations.** (F-003, BR-002, BR-006)
4. **ADR — Decide on-chain PII minimization and the credential retention / RA 10173 erasure stance
   given on-chain permanence.** (BR-002, BR-006; data classification)
5. **ADR — Decide the credential lifecycle/status model (append-only vs revocable/updatable),
   enabling a revocation path for compromised or superseded credentials.** (open question; relates
   to F-101 future scope)
